**Feature Branch (功能分支)**: `001-rag-advisor-match`  
**Created (建立日期)**: 2026-03-30

**Input (使用者輸入)**: "本 MVP 旨在驗證「使用者以自然語言描述需求 → 系統自動媒合理專並提供可解釋推薦」的可行性。流程上，首先將 10 份 Word 理專文件解析為結構化 JSON（欄位包含專業領域、熟悉客群、溝通類型等）與完整文本描述兩種資料形式；結構化資料用於條件匹配，文本用於語意檢索與生成解釋。查詢時，LLM 先將使用者輸入轉換為標準化需求（如退休規劃、高資產客群、溫和溝通），再以 embedding 對文本做 semantic search 取得候選理專，並結合欄位比對進行加權 scoring（專業領域、客群、溝通風格等）完成排序。最後將 Top N 理專與其關鍵匹配特徵交給 LLM 生成推薦結果與理由，同時回傳對應欄位或文本片段作為依據，確保可解釋性。整體系統由文件處理、向量索引、查詢解析、排序邏輯與生成層組成，前端提供輸入需求、展示推薦結果與來源依據的簡單介面，即可完成一個可互動的 RAG + matching Demo。"

## Clarifications
### Session 2026-03-30
- Q: 當使用者的自然語言需求無法與任何現有理專的資料媒合時，系統該如何處理？ → A: 放寬評分條件，回傳部分符合 (Partial Match) 的理專，並在理由中明確標示「有哪些條件未吻合」。
- Q: 當使用者輸入模糊不清的內容，導致 LLM 無法解析為標準化需求時，系統該如何應對？ → A: 略過結構化條件比對，完全依靠對理專文本描述的「純語意搜尋 (Semantic Search)」來產生推薦。
- Q: 解析 Word 文件時若遇到欄位資料缺漏，應該如何處理？ → A: 寬容處理，缺失欄位填寫「未提供 (Not Provided)」，保留該理專的文本以供語意搜尋，且不排除進入排名。

## User Scenarios & Testing (使用者場景與測試) *(必填)*

### User Story 1 - 透過自然語言搜尋理專 (Priority: P1)

使用者可以使用自然語言輸入他們的財務需求，並獲得一份推薦的理專清單。系統會提供易於理解的解釋，說明使用者的需求如何與這些理專的資格相符。

**Why this priority (為何設定此優先級)**: 驗證「從自然語言轉換到合理的推薦結果」這個核心流程是本次 MVP 的首要目的。

**Independent Test (獨立測試方式)**: 可透過 UI 介面送出一句一般的使用者查詢，並驗證系統是否能（從初始化的 10 份文件知識庫中）回傳適當的理專，以及相對應的文本解釋說明。

**Acceptance Scenarios (驗收場景)**:

1. **前提 (Given)** 10 份 Word 文件皆已被解析並建立索引 (indexed)，**當 (When)** 使用者輸入類似「我在尋找一位能為高資產客群進行退休規劃，且溝通風格溫和的理專」的查詢時，**則 (Then)** 系統會顯示最符合的理專，並引用結構化欄位（例如：溫和溝通）以及非結構化文本來解釋推薦理由。
2. **前提 (Given)** 系統正常運行中，**當 (When)** LLM 產生理專推薦理由時，**則 (Then)** 這些理由必須包含原始 Word 文件或解析出的 JSON 條件的明確引用/片段 (snippets)。

---

### Edge Cases (邊緣情況)

- **查無完全符合者**：當自然語言需求無法與現有理專資料完全吻合時，系統將放寬評分條件，回傳部分符合 (Partial Match) 的候選理專，並在生成的理由中明確標示「有哪些使用者特定條件未達成」，避免死胡同。
- **輸入過於模糊、難以解析標準化需求**：略過結構化條件比對階段，直接針對使用者的原文與理專的文字描述進行「純語意搜尋 (Semantic Search)」，利用 RAG 系統對未結構化文本的理解力來產生推薦結果。
- **文件解析欄位缺失**：若 Word 文件中缺少特定結構化欄位（如未註明溝通風格），系統將賦予該欄位「未提供 (Not Provided)」之預設值。這表示在該特定維度的硬性比對中可能得分較低，但該理專仍可透過非結構化文本的語意搜尋被找到並推薦。

## Requirements (需求) *(必填)*

### Functional Requirements (功能需求)

- **FR-001**: 系統必須將包含理專資料的 10 份 Word 文件，解析為結構化的 JSON（專業領域、客群、溝通類型）以及非結構化的完整文本描述。
- **FR-002**: 系統必須使用 LLM 將使用者的自然語言輸入解析為標準化的需求欄位。
- **FR-003**: 系統必須針對非結構化文本描述執行語意搜尋 (semantic search)，並利用 embeddings 來尋找候選理專。
- **FR-004**: 系統必須利用「結構化欄位比對」以及「語意搜尋 (semantic search) 關聯度」的加權組合，對所有的候選理專進行評分 (scoring) 與排序。
- **FR-005**: 系統必須使用 LLM，為排名前 N 名 (Top N) 的理專產生自然語言的推薦理由 (rationales)。
- **FR-006**: 系統必須回傳原始文本片段 (snippets) 或欄位值作為憑據，以確保推薦結果具備可解釋性 (explainability)。
- **FR-007**: 系統必須提供一個簡單的前端介面 (frontend)，用於接收查詢並渲染出推薦清單與其參考來源證據。

### Key Entities (關鍵實體) *(如涉及資料則必填)*

- **Advisor Profile (結構化理專資料)**: 代表理專的分類資料（專業領域、目標客群、溝通類型）。
- **Advisor Profile (文本理專資料)**: 理專的完整對話/描述性呈現，用於語意搜尋 (semantic search)。
- **User Query (使用者查詢)**: 自然語言輸入與其解析/標準化後的參數。
- **Recommendation Result (推薦結果)**: 結合了理專資料、媒合分數 (score)、AI 生成的推理理由，以及來源引用 (citations) 的聚合輸出。

## Success Criteria (成功標準) *(必填)*

### Measurable Outcomes (可量化成果)

- **SC-001**: 系統成功處理並索引 (index) 100% 提供的 10 份 Word 文件，並轉換為有效的 JSON 和文本元件。
- **SC-002**: 在每次查詢中，使用者能在 15 秒內收到排名前 N 名 (Top N) 的理專推薦清單及生成的解釋。
- **SC-003**: 100% 的推薦結果至少包含一個可驗證的軌跡/引用 (citation)，能連結回原始來源文本或 JSON 欄位。

## Assumptions (假設前提)

- 提供的 10 份 Word 文件具有相對一致的結構或充足的上下文，能確保可靠地提取出詮釋資料 (metadata) 欄位（專業領域、客群、溝通風格）。
- 採用標準的現成模型 (off-the-shelf models) 而不經過客製化微調 (fine-tuning)，即能達到可接受的 embeddings 品質與 LLM 生成效能。
- 身分驗證 (Authentication)、使用者工作階段 (user sessions) 及永久的查詢歷史紀錄不包含在本次 MVP 的範圍內。
